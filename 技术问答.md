# MusicMap 技术问答（答辩准备）

> 每题结构：**一句话先答**（被问到先甩这句）→ **展开**（讲清原理）→ **可能追问**（评委往下挖时的弹药）。
> 关键数字：全量 **48.9 万张封面 / 223 国**；**8 张 8192×8192 图集**（每张 256×256=6.5 万格、每格 32px，共 52.4 万容量）；显存约 **2GB**；预烤产物 **≈190MB**。

---

## Q1. 这些图片为什么能加载得这么快？

**一句话先答：** 因为我们不在浏览器里逐张拉几十万张原图，而是后端预先把所有封面拼成 8 张大图（图集），前端只下这 8 张 + 一份清单，一次性传上 GPU。

**展开（四层叠加）：**

1. **GPU 实例化 + 纹理图集**：所有封面共用**一个 mesh、一次 draw call**。封面像素全预拼进大图集，运行期只改每个实例的 UV 坐标和出现时间，不再"往场景里加图片节点"。原型实测 10 万张带动画 **92fps**。
2. **预烤图集（核心）**：把"拼图"从浏览器搬到后端只跑一次。前端若实时拼，要逐张拉 48.9 万张 640px 原图 ≈ **24GB**，量级远超浏览器缓存配额，关机就被清、无法秒开。预烤后前端只下 **8 张图 + manifest ≈190MB**，**任何浏览器、任何机器、重启后缓存命中即秒开**。
3. **多图集突破单纹理上限**：WebGL 单张纹理最大 16384²。我们用 32px/格、8192² 一张装 6.5 万张，挂 8 张 = 52.4 万容量，装下全部 48.9 万；着色器按实例的 `iAtlas` 选 `uA0..uA7` 采样。
4. **增量上传防黑屏**：8 张共约 2GB 纹理若一帧内全灌显存，单次 GPU 上传超 2s 会触发 Windows TDR（显卡超时重置）→ 丢 WebGL 上下文 → 永久黑屏。改成每张加载完立刻 `initTexture` 单独上传、帧间 `requestAnimationFrame` 让一帧。

**可能追问：**

- *"190MB 还是不小？"* → 它是一次性、全局共用的成品，且带 immutable 长缓存；第二次打开是 0 网络。相比实时 24GB 是两个量级。
- *"显存 2GB 哪来的？"* → 8 × 8192² × 4 字节(RGBA) ≈ 2GB。JPEG 只省下载和磁盘，上 GPU 都会解成未压缩 RGBA，所以省不掉，只能控制上传节奏。

## Q2. 后端"烤图"是干什么的？预处理怎么做？

**一句话先答：** "烤图"就是把散落在对象存储里的几十万张封面，离线拼成几张大图集 + 一份索引清单，让前端不用自己拼。脚本是 [bake_atlas.py](backend/app/bake_atlas.py)。

**展开（流程）：**

1. **查候选**：从 PostGIS 查出全部 `cover_status='downloaded'` 的封面，按 **国→年→mbid 确定性排序**（港澳台并入 CN）。排序固定很关键——格子顺序由它决定。
2. **逐张读 + 缩**：从 MinIO 对象存储按 `{mbid前2位}/{mbid}.jpg` 读出原图，缩到 **32×32**，用线程池 32 并发跑。
3. **拼图集**：第 g 张图集固定装第 `[g·65536, (g+1)·65536)` 个候选，`paste` 到 8192² 画布的对应格子，存成 JPEG(q85)。
4. **写清单 manifest.json**：记录版本号、格子尺寸、图集数，以及
   - `pool`：每个国家的 `[年份, mbid, 曲风码]` 列表，**遍历顺序 = 格子顺序**，前端据此还原 `mbid → 第几张图集的哪个格子`；
   - `counts`：每年每国封面数，给时间轴动画定节奏。

**预处理的几个工程点（可主动讲，显得严谨）：**

- **断点续烤**：因为格子号=候选序号是确定的，已写出的图集直接跳过，被 Docker 重启掐断后重跑只补缺的。
- **封面入库前也有预处理**（在 ETL 阶段，[phase2_download_covers.py](etl/phase2_download_covers.py)）：从 CAA 下载后统一 **等比缩到 640²、白底居中、转 JPEG**；并做**炸弹图防护**——超 5000 万像素的图直接拒，避免 PIL 解一张图吃几个 GB 内存触发 OOM。
- **版本号失效**：manifest 带 `version`，前端取图带 `?v=version`，重烤后缓存自动失效。

**可能追问：**

- *"为什么不直接让前端拉原图？"* → 见 Q1 第 2 点，24GB vs 190MB、且缓存留不住。
- *"封面库更新了怎么办？"* → 重跑一次烤图脚本，version 变，前端 immutable 缓存自动换新。

## Q3. 这个 3D 地球是怎么做的？

**一句话先答：** 用 three.js 手绘的矢量地球——海洋是一个青绿色球，陆地国界画进一张大 canvas 贴上去，封面则是 GPU 实例化的小平面，出现/缩放动画全在着色器里算。代码在 [MapGlobe3D.tsx](frontend/src/components/MapGlobe3D.tsx)。

**展开（分层）：**

- **底图**：海洋 = 一个青绿球体；**陆地/国界** = 把数据库里的国界多边形画进一张 8192×4096 的等距圆柱(equirectangular) canvas（粉白填充 + 浅粉辉光描边），再贴到一个和经纬严格对齐的 UV 球上。外加边缘 fresnel 白辉光、星空点、以及用 CSS2D 渲染的国家/大洲/海洋标注（背对镜头的半球自动隐藏，字号随相机距离缩放）。
- **封面层**：`InstancedMesh` + 自定义 **billboard 着色器**。每个网格坑位一个实例，实例属性 `iPos / iUV / iBorn / iAtlas`，配 uniform `uTime` → **出现、缩放(回弹缓动)、替换动画全在顶点着色器里算**，运行期 CPU 只改少量属性、零重排。
- **国界极光光幕**：沿每段边界向外径向拉出竖直 ribbon，绿↔暖橙按经度渐变 + 微流动，让封面盖住地表后各国轮廓仍浮在上层。
- **交互**：OrbitControls 旋转缩放；点击用 raycaster 打到地球，找最近的"已点亮"封面取元数据 → 弹 Spotify 预览；按下到松手位移 >6px 判为拖拽旋转、不触发搜索。
- **省电细节**：scroll story 不在地图页时（场景层无 `is-active`）直接跳过渲染循环，省 GPU。

**可能追问：**

- *"为什么不用现成的 mapbox/deck.gl？"* → 见 Q5。
- *"动画为什么不卡？"* → 动画逻辑全在 GPU 着色器，CPU 每帧只改极少的实例属性，没有 DOM/图层重排。

## Q4. 数据从哪来？MusicBrainz 和 CAA 是什么，为什么用它们？

**一句话先答：** 专辑元数据来自 **MusicBrainz**（开放的全球音乐数据库），封面图来自 **Cover Art Archive(CAA)**；两者用同一个 `mbid` 关联，天然对得上。

**展开：**

- **MusicBrainz**：一个开源、开放许可的音乐元数据库（类似"音乐界的维基百科"）。每个发行有一个全球唯一稳定的 ID——`mbid`。我们的 `app.albums` 存 `mbid / 标题 / 主艺人 / 发行年 / 艺人国家 / cover_status`，只取 2010–2025、有国家、封面已下载的。
- **Cover Art Archive(CAA)**：MusicBrainz 的官方配套封面库，按 `release-group/{mbid}/front` 就能取封面。ETL 阶段按 mbid 从 CAA 下载，缩成 640² JPEG 存进 **MinIO** 对象存储。当前已下 **48.9 万张 / 223 国**。
- **地理底图 = PostGIS**：`app.countries`（真实国界 + 质心/大洲/面积）、`app.country_grids`（每国铺的网格坑位，约 5–6 万个，即"封面能冒出来的位置"）。

**为什么选它们（这点和项目立意强相关，建议讲透）：**

1. **同一个 mbid 串起元数据和封面**——MusicBrainz 出身份、CAA 出图，一把钥匙开两把锁，join 干净，不用跨源对名字。
2. **不被榜单/流量偏置**——这正是我们的立意。Spotify 这类源是按播放量/市场排的，会放大欧美热单、埋没印度巴西这些高产但"不上榜"的场景；MusicBrainz 是**以"发行"为单位的全量登记**，才能回答"剥离算法后，世界到底在创作什么"。
3. **开放、可复现、免费、覆盖广**——开放许可、覆盖小众与非英语市场，适合做学术性可视化；不像商业 API 那样强限流、强商用限制。

**可能追问：**

- *"那 Spotify 用在哪？"* → 只用在点击封面后的 30 秒试听预览（锦上添花），不作为数据主干；而且它限流很严，不适合做全量。
- *"数据完整吗 / 会不会有缺图？"* → CAA 不是每个 release 都有封面，缺图的格子留一格背景色，不影响整体；`cover_status` 字段记录每张的下载状态，可断点续下、只重试失败的。
- *"为什么把港澳台并入 CN？"* → 它们在底图里没有独立网格坑位，做了归并以对齐网格，纯属技术处理。

## Q5.（高频追问）为什么不用现成的地图库？

**一句话先答：** 试过，maplibre 和 deck.gl 都撑不住"几十万张带动画的封面"。

- **maplibre**：动态 `addImage` 加符号会触发图层重排，一加图就掉到 **14fps**；globe 投影下符号层还有限制。
- **deck.gl**：`IconLayer` 在 globe 投影下根本不渲染（只有散点能用），满足不了封面需求。
- **three.js**：对 GPU billboard、着色器、纹理完全可控，才能做到"几十万张带动画满帧 + 自定义出现/替换"。底图精度无所谓，所以自绘矢量球就够。

## Q6. 前后端是怎么组织的？Docker 怎么实现前后端分离？

**一句话先答：** 我们做的是彻底的前后端分离——后端用 Docker Compose 把 FastAPI 和它依赖的 PostGIS、Redis、MinIO 编排成一组容器，彼此用服务名在内部网络互连、只对外暴露 8000 端口的纯 JSON/资源 API；前端是独立的 React+Vite 单页应用，跑在 Docker 之外，两端只通过 `/api` 契约耦合。

**展开（整体拓扑）：**

```
┌─────────────────────────────────────────────────────────┐
│  Docker (docker-compose 编排，内部网络互通)                  │
│   postgis:5432 ──┐                                        │
│   redis:6379  ───┼──►  backend (FastAPI/uvicorn :8000)    │
│   minio:9000  ───┘         ▲                              │
└───────────────────────────┼──────────────────────────────┘
                            │ HTTP /api/*  (仅 JSON + 图集 jpg)
                   frontend (Vite/React :5173)  ← 在 Docker 之外，宿主机直接跑
```

compose 里只有四个服务：**postgis、redis、minio、backend**（`docker-compose.yml`）。前端是独立 Vite/React 工程，不进容器。

**"分离"分在两层：**
1. **技术栈/进程分离（真正的前后端分离）：** 后端是纯 API，连根路由 `/` 都只返回一段 JSON（`main.py` 的 `root()`），**不托管 React 页面**（只有 debug 用的 `/inspector`）。前端是纯 SPA，构建成 `frontend/dist/` 静态文件。两边唯一耦合是 `/api/*` 契约，前端只 `fetch('/api/...')`，不关心后端是 Python 还是别的。
2. **Docker 打包"后端 + 基础设施"：** Docker 在这里分离的是后端服务与它依赖的数据库/缓存/对象存储。

**Docker 具体靠三个机制实现：**
1. **服务名当 DNS——容器间靠名字互连。** 后端连库不用 `localhost`，而是服务名（`docker-compose.yml`）：`postgis:5432` / `redis:6379` / `minio:9000`。compose 默认建一个 bridge 网络，容器在里面用服务名解析，谁的端口都不暴露给宿主机也能内部通。
2. **只把该露的端口映射出去。** `backend` 映 `8000:8000`，浏览器才够得着；postgis/redis 虽也映了端口（方便 DBeaver 直连），但业务上前端只跟 8000 说话。
3. **启动顺序 + 热重载。** `depends_on: postgis condition: service_healthy` 等数据库 healthcheck 过了后端才起；`./backend/app:/app/app` bind mount + `uvicorn --reload`，改 .py 不用重 build。

**前端怎么跨过"分离"连后端（两道保险）：**
- **开发期走 Vite 代理**（`vite.config.ts`）：前端 `fetch('/api/xxx')` 被转发到 `http://127.0.0.1:8000`，浏览器眼里同源，没有跨域。注意特意用 `127.0.0.1` 不用 `localhost`——Windows 下 localhost 先解析 IPv6 `::1`，Docker 端口转发偶发在 `::1` 挂起导致代理超时。
- **后端兜底开 CORS**（`main.py` `allow_origins=["*"]`）：即便前端不走代理直接跨域请求也放行。

**可能追问：**
- *"前端为什么不也放进 Docker？"* → 开发期前端用 Vite dev server 热更新更快，没必要容器化；目前是"后端容器化 + 前端裸跑/静态"的半分离。要完整容器化，加个多阶段 build + nginx 的 frontend Dockerfile 和对应 compose 服务即可。
- *"生产怎么部署前端？"* → `vite build` 出 `dist/` 静态文件，用任意静态服务器/CDN/nginx 托管，反代把 `/api` 指向后端 8000。
- *"为什么连库用服务名而不是 IP？"* → 容器 IP 每次重启会变，compose 内置 DNS 把服务名解析到当前容器，稳定且可读。

## 一页速记（上台前扫一眼）

- **快**：后端预烤 8 张图集（190MB），GPU 一个 draw call、UV 全在着色器算；实时拼要 24GB。
- **烤图**：DB 查候选(国→年→mbid 定序) → MinIO 读图缩 32px → 拼 8192² → 写 manifest(pool=格子顺序, counts=节奏)；可断点续烤。
- **地球**：three.js 手绘矢量球（海洋球 + 国界 canvas 贴图）+ 实例化封面 + 着色器动画 + 极光边界。
- **数据**：MusicBrainz(元数据/mbid) + CAA(封面，同 mbid) + PostGIS(国界/网格)；选它们因为开放、可复现、且不被榜单偏置——契合"剥离算法看真实创作"。
- **黑屏坑**：2GB 纹理别一帧灌完，逐张 initTexture + 帧间让出，否则触发 TDR。
- **架构**：前后端分离。compose 编排 postgis/redis/minio/backend 四容器，靠服务名内网互连、只露 8000 纯 API；前端 React+Vite 在 Docker 外，dev 走 vite 代理(127.0.0.1:8000)避跨域，只靠 `/api` 契约耦合。
